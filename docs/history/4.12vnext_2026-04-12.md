# NDX Agent 架构争论与"三层真实"架构决议

> **存档时间**: 2026-04-12
> **参与方**: 用户（项目所有者） vs Claude（AI 助手）
> **主题**: vNext 重构方向、Subagent 的本质、以及到底应该用什么样的架构实现五层投资分析框架
> **性质**: 这是一份记录从错误认知到正确认知的完整辩论档案。

---

## 一、vNext 试图解决的真实问题

### 1.1 Legacy `analyzer.py` 的三重瓶颈

| 瓶颈类型 | 具体表现 | 对投资分析质量的影响 |
|---------|---------|-------------------|
| **物理性** | 单个 Prompt 塞入 L1-L5 全部 40+ 指标 + NDX_COMMAND_V9 + 4C 范例 + 推理范例 | Token 超载，模型注意力被迫平均分配 |
| **结构性** | Analyst → Critic → Reviser 三段式黑盒 | 无法验证"L3 AD Line 恶化如何具体影响 L5 趋势判断" |
| **认知性** | 跨层关系只是"顺带一提" | 五层框架的核心——层间支撑/冲突——被淹没在指标罗列中 |

### 1.2 vNext 的设计意图：显式认知分工

vNext 的目标是把隐含的大脑内部分工变成**显式的计算步骤**：

```
L1 Analyst ──┐
L2 Analyst ──┼→ Bridge Agents → Thesis Builder → Critic/Risk/Schema → Reviser → Final Adjudicator
L3 Analyst ──┤    (显式桥接)        (显式整合)        (显式审查)        (显式修订)     (显式裁决)
L4 Analyst ──┤
L5 Analyst ──┘
```

**这一步设计完全正确。**

### 1.3 vNext 当前的致命伤

`feature/vnext-subagent-rebuild` 分支的问题不是设计方向错了，而是**执行顺序反了**：

- 在还没有验证"真实的分层 LLM 调用是否能产生更高质量分析"之前
- 就已经写了近 3000 行框架代码（`contracts.py` ~771 行、`validators.py` ~584 行、硬编码 stub 的 `orchestrator.py` ~684 行）
- `orchestrator.py` 的核心 LLM 调用步骤至今返回的是写死字典

**结论**: vNext 是在用 Python 代码模拟一个 LLM orchestrator，而不是直接让 LLM 工作。

## 二、关于 Subagent 的认知进化史

### 2.1 第一阶段（错误）：Claude 的初始理解

Claude 最初认为：
> "`.claude/agents/` 只能在交互式对话中手动 `@` 触发，不能替代自动化流水线。"

这是**不完整的**。

### 2.2 第二阶段（修正）：读了官方文档后的理解

读了 [code.claude.com/docs/zh-CN/sub-agents](https://code.claude.com/docs/zh-CN/sub-agents) 和 [headless](https://code.claude.com/docs/zh-CN/headless) 文档后，Claude 修正了理解：

1. **Subagent 可以后台并发运行** (`background: true`)
2. **Headless 模式 (`claude -p`) 支持程序化调用 subagents**，可以通过 `--agents` CLI 标志内联定义角色
3. **Subagent 不能嵌套**——这是硬约束。Layer Analyst 不能自己调用 Bridge Agent

但 Claude 此时走向了另一个错误：
> "那我们可以用 `claude -p` + subagents 来构建自动化流水线。"

### 2.3 第三阶段（被用户痛击后）：对"2-3 个 API 调用"的自我否定

用户尖锐质问：
> "精简后，可能只需要 2-3 个 API 调用——这是否是放屁？"

**答案是：是的，那句话是放屁。**

Claude 当时暗含的方案是：
- Call 1: "L1+L2+L3 打包分析师"
- Call 2: "L4 估值分析师"
- Call 3: "Bridge + Final 综合裁决"

这本质上**把多个异质分析职责重新塞回一个 Prompt**，退回到了 legacy 架构的同一个坑里。官方文档反复强调：

> *"每个 subagent 应该在一个特定任务中表现出色"*
> *"设计专注的 subagents"*

把 L1（宏观流动性）和 L3（市场广度）打包给一个模型，等于要求它同时戴两副完全不同的专业眼镜，这必然导致注意力稀释和视角污染。

### 2.4 最终正确理解

**Subagent 的本质是"上下文隔离器 + 角色专用容器"**：
- 每次调用启动一个只接收专属 system prompt 的隔离上下文
- 主对话历史不会继承
- 核心价值不是"省钱"，而是"买质量 + 买可控"

**`.claude/agents/` 与自动化的关系**：
- `.claude/agents/` 是**角色定义模板**
- 自动化执行需要配合 `claude -p` (headless) 或 Agent SDK
- 但 `claude -p` 每次都要启动完整 Claude Code 运行时，对"每日定时报告"场景有明显启动开销

**Agent-Teams 的定位**：
- 实验性功能，默认禁用
- 队友之间可以直接通信、辩论
- 适合"委员会式"协作，不适合"流水线式"处理
- **对 NDX Agent 是过度设计**

## 三、辩论的核心交锋点

### 交锋 1: vNext 的 3000 行 Python 框架是否必要？

**用户立场**: 怀疑现有架构低效、错误，不如直接用 subagent 搞定。
**Claude 初始立场**: 大幅瘦身即可，不必推倒重来。
**修正后共识**: 不是"大幅瘦身"，而是**彻底废弃无意义的框架层**，保留正确的分析步骤分离，但用极简代码实现。

### 交锋 2: 到底需要多少次 LLM 调用？

**Claude 错误主张**: 2-3 次（打包 Layer）。
**用户反驳**: 这是混合职责，是放屁。
**修正后结论**: **5-7 次真实调用**。L1-L5 各 1 次（并行），Bridge 1 次，Final Adjudicator 1 次。不压缩，不打包。

### 交锋 3: `.claude/agents/` 应该扮演什么角色？

**Claude 初始主张**: 不适合自动化，只做交互增强。
**修正后主张**: 确实不适合作为核心自动化流水线（因为启动开销和不能嵌套），但可以作为**交互式解释增强层**。这个判断经受住了官方文档的检验。

## 四、最终决议："三层真实"（Three-Layer Real）架构

### 4.1 架构总览

```
┌─────────────────────────────────────────────────────────────────┐
│  Layer 1: 数据与预处理（Python，稳健、自动化、每日运行）            │
│  ├── tools_L1.py ~ tools_L5.py（已成熟）                         │
│  ├── DataCollector → data_json                                   │
│  └── packet_builder.py → AnalysisPacket（保留，但只负责数据整理）  │
└──────────────────────────┬──────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────────┐
│  Layer 2: 认知流水线（Python 轻量 orchestrator，真实 LLM 调用）    │
│  ├── 5 个独立的 Layer Prompts（直接调用 LLMEngine）              │
│  ├── 1-2 个 Bridge/Thesis Prompts（接收 LayerCards，输出整合）    │
│  ├── 1 个 Final Adjudicator Prompt（独立裁决）                    │
│  └── 极简的 jsonschema 校验 + 重试                               │
│                                                                  │
│  关键：这里不需要 contracts.py 帝国，只需要 prompt 模板 +          │
│        LLMEngine.call_with_fallback() + json parse。             │
└──────────────────────────┬──────────────────────┬───────────────┘
                           │                      │
                           ▼                      ▼
              ┌────────────────────┐   ┌────────────────────┐
              │  legacy_adapter.py │   │  output/agent/     │
              │  → ReportGenerator │   │  中间产物存档       │
              └────────────────────┘   └────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────────┐
│  Layer 3: 交互增强（.claude/agents/，按需手动触发）                │
│  ├── ndx-l1-explainer.md：解释宏观数据的深层含义                  │
│  ├── ndx-bridge-explainer.md：解释跨层冲突的投资意义              │
│  └── ndx-critic.md：对已有分析报告进行深度质疑                    │
│                                                                  │
│  这些不是自动化的一部分，而是"当用户对某个结论存疑时，            │
│  我可以 @ 调用它来提供一个比主报告更深入的解读"。                  │
└─────────────────────────────────────────────────────────────────┘
```

### 4.2 各层详细设计

#### Layer 1: 数据与预处理

**保留的代码**：
- `tools_L1.py` ~ `tools_L5.py`：数据函数已成熟，命名规范统一
- `DataCollector`：自动化数据收集流水线
- `packet_builder.py`：但职责收窄为"数据整理 + 层状态判断"，删除硬编码的 `CROSS_LAYER_RULES`

**删除或简化的代码**：
- `contracts.py` 的 771 行 Pydantic 帝国 → 用 `TypedDict` 替代
- `validators.py` 的 584 行自定义校验框架 → 用 `jsonschema`（每个 prompt 20 行以内 schema）+ 重试替代

#### Layer 2: 认知流水线

**调用矩阵（质量优先原则）**：

| 步骤 | 输入 | 输出 | 推荐模型 |
|------|------|------|----------|
| L1 Analyst | L1 宏观数据 | LayerCard | Sonnet |
| L2 Analyst | L2 情绪数据 | LayerCard | Sonnet |
| L3 Analyst | L3 广度数据 | LayerCard | Sonnet |
| L4 Analyst | L4 估值数据 | LayerCard | **Opus**（最复杂） |
| L5 Analyst | L5 技术数据 | LayerCard | Sonnet |
| Bridge Agent | 5 张 LayerCards | BridgeMemo（显式冲突矩阵） | Opus 或 Sonnet |
| Final Adjudicator | BridgeMemo + Thesis + Risks | FinalAdjudication | **Opus** |

**关键设计决策**：
1. **Bridge 和 Thesis 不能合并**：Bridge 的职责是客观识别冲突，Thesis 的职责是主观整合论点。合并会导致"和稀泥"。
2. **Final Adjudicator 必须独立**：裁判不能兼运动员。
3. **`orchestrator.py` 目标代码量**：从 684 行压缩到 **150-200 行**。没有 stub，只有真实 LLM 调用。

**向后兼容**：
- `legacy_adapter.py` 必须保留，把 5-7 步产物映射回 `__LOGIC__` JSON，供现有 `ReportGenerator` 消费。

#### Layer 3: 交互增强

这不是自动化的一部分，而是自动化的"放大镜"。当用户对报告结论有疑问时，可以通过 `@ndx-l1-explainer` 等方式获得更深一层的解释。这与官方文档对 subagent 的定义完全吻合：

> *"任务产生您不需要在主上下文中的详细输出"*
> *"工作是自包含的，可以返回摘要"*

### 4.3 第一性原理支撑

| 原则 | 架构回应 |
|------|---------|
| **质量优先** | 不压缩调用次数，Bridge 和 Final 用最强模型 Opus |
| **可解释性** | 每步都有明确产物（LayerCard、BridgeMemo、FinalAdjudication），可追踪 |
| **低门槛修改** | 投资规则写入 prompt 模板和 `src/纳斯达克100指挥中心[MECE].txt`，修改不需要改 Python 框架 |
| **自动化优先** | 核心流水线是纯 Python，不依赖 `claude -p` 启动开销 |
| **向后兼容** | `legacy_adapter.py` 保护现有报告系统 |

## 五、被明确否定的事物

以下是在本次辩论中被明确否定的方案：

1. **❌ 继续堆 vNext Python 框架**（stub 帝国、contracts 帝国、validators 帝国）
2. **❌ 把 5-7 个分析步骤压缩成 2-3 个大 Prompt**（混合职责，回归 legacy 陷阱）
3. **❌ 用纯 `claude -p` + subagents 替代核心自动化流水线**（启动开销大，且不能嵌套）
4. **❌ 使用 Agent-Teams 作为核心架构**（实验性功能，对流水线场景是过度设计）
5. **❌ 在 Python 里继续手写一个次等的 Agent Orchestrator**（Claude Code 运行时已经免费提供了这些能力，但你不应该把自动化流水线绑定到交互式运行时上）

## 七、后记：这场辩论的价值

这场辩论的核心价值在于，它暴露了 AI 助手在给出架构建议时的一个典型陷阱：

> **为了"可控性"和"工程优雅"，不自觉地把问题压缩回更少的抽象层，从而牺牲了解决问题本身所需的真实复杂度。**

用户的尖锐质疑（"这是否是放屁？"）迫使 Claude 从第一性原理重新审视：
- 不是"怎么让代码更少"
- 而是"怎样的架构能产生最高质量的投资分析"
- 在质量确立后，才考虑如何用最少、最清晰的代码去实现它

**"三层真实"的本质，是拒绝一切假装和 stub，让每一层都在做它真正该做的事。**
