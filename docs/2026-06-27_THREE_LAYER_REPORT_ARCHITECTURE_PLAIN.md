# 三层研报架构总指挥：纯数据、新闻事件、综合总报告

更新日期：2026-06-27

本文是 `ndx_vnext` 后续“三层研报架构接入攻坚”的总指挥文件。

它不是普通说明稿，而是回答一个核心问题：

> 如何让 NDX 研报既保持正式数据链干净，又能理解真实世界事件，最后生成一份可读、可审计、可反驳的综合总报告？

一句话结论：

> 保留现有 vNext 纯数据研报；新增独立新闻/事件简报；再由综合总报告读取前两者，做交叉质询、分级解释和风险裁决。三份报告独立生成、独立落盘、独立入口、互相引用，但不得互相污染。

---

## 1. 架构总原则

### 1.1 三层不是排版分类，而是证据治理

三层研报不是为了多做几个页面，而是为了把三种性质完全不同的东西分开：

| 层级 | 它是什么 | 它回答什么 | 它不能做什么 |
| --- | --- | --- | --- |
| 第一层：纯数据研报 | 正式数据链 | 正式数据现在显示什么 | 不能接收新闻、浏览器材料、市场叙事来改写 L1-L5 |
| 第二层：新闻/事件简报 | 外部世界事件账本 | 发生了什么、市场正在讲什么故事 | 不能冒充 L1-L5 evidence refs |
| 第三层：综合总报告 | 数据与新闻的交叉质询 | 数据和新闻如何互相确认、削弱、冲突或暂时无法解释 | 不能用顺滑叙事抹平冲突，不能用单条新闻证明复杂市场因果 |

通俗说：

- 纯数据研报像体检报告。
- 新闻/事件简报像当天病史和外部刺激记录。
- 综合总报告像医生会诊意见。

医生可以综合判断，但不能把传闻写成化验结果，也不能为了安慰病人而隐藏异常指标。

### 1.2 最底层金融学逻辑

指数研究的第一性原理很简单：

> 指数价格是市场对未来现金流、贴现率、风险溢价、资金条件、持仓结构和交易情绪的综合定价。

所以，一份 NDX 研报必须同时回答五个问题：

1. 我们到底在判断什么对象？
2. 当前价格反映了什么预期？
3. 哪些正式数据支持这个判断？
4. 哪些事件和新闻可能解释市场变化？
5. 什么证据会推翻当前判断？

只看数据，会干净但可能慢。

只看新闻，会敏感但容易乱。

只看最终结论，会好读但不可审计。

三层架构的意义，就是把这三个问题拆开处理，再谨慎合成。

### 1.3 顶级研究机构的共同思路

顶级投行和资管机构研究纳指、标普或其他权益指数时，通常不会只看一个指标，也不会只讲新闻故事。它们常见的底层框架包括：

- 宏观与利率：政策利率、长端利率、真实利率、通胀补偿。
- 盈利与估值：盈利增长、利润率、PE、forward PE、FCF yield、ERP。
- 信用与流动性：信用利差、融资条件、美元流动性。
- 市场结构：集中度、广度、行业贡献、权重股驱动。
- 波动与风险偏好：VIX/VXN、期限结构、尾部风险定价。
- 新闻与事件：央行、财报、监管、财政、地缘、产业周期。

真正专业的地方不在于“指标多”，而在于：

- 每个指标知道自己能证明什么。
- 每条新闻知道自己只是事件材料还是强证据。
- 每个结论都知道自己的反证条件。

三层架构正是把这套研究纪律产品化。

## 2. 三条不可破坏的红线

### 红线一：纯数据层必须保持干净

第一层纯数据研报继续使用现有 vNext 主链：

- L1 宏观、利率、流动性。
- L2 风险偏好、信用、波动。
- L3 指数内部健康、广度、集中度。
- L4 估值、盈利质量、风险溢价。
- L5 价格趋势、成交、技术状态。
- Bridge / Thesis / Risk / Final。
- DataIntegrity、Prompt Inspector、Run Review。

它只能消费正式数据源和已经升级为正式数据链的 evidence refs。

它不能消费：

- 新闻摘要。
- 浏览器页面。
- 登录态工具结果。
- 市场叙事。
- sidecar 候选材料。
- 综合总报告的任何判断。

原因很简单：

> 如果新闻先进入 L1-L5，模型很容易先相信一个故事，再反过来解释数据。这样纯数据研报就不再是纯数据研报。

### 红线二：新闻层只能做事件账本，不能冒充正式证据

第二层新闻/事件简报可以整理外部世界发生了什么，但它不是第一层的证据源。

新闻层可以说：

- Fed 发生了什么。
- 公司披露了什么。
- 主流媒体报道了什么。
- 市场正在关注什么叙事。
- 这件事可能影响哪条金融链路。
- 这件事有哪些限制和不确定性。

新闻层不能说：

- 因为某条新闻，所以 L1-L5 应该改结论。
- 因为媒体说市场乐观，所以信用风险不存在。
- 因为某家公司财报好，所以整个 NDX 内部健康。
- 因为一条 AI 新闻，所以估值压力解除。

新闻可以提供解释线索，但不能自动升级为正式证据。

### 红线三：综合层可以解释，但必须分级、可追溯、保留反证

第三层综合总报告是最有价值、也最危险的一层。

它可以做解释，但每个解释必须标明：

- 这是数据支持的判断，还是新闻支持的叙事？
- 这是两边共同支持，还是只是合理假设？
- 反驳证据是什么？
- 未解决的张力是什么？
- 什么情况会让这个解释失效？

综合层不能为了读起来顺滑而抹平冲突。

如果数据和新闻无法互相解释，就必须写：

> 暂时无法解释。

这不是失败，而是研究诚实。

## 3. 第一层：纯数据研报

### 3.1 定位

纯数据研报是现有 vNext 主产品。

它的任务是：

- 看正式数据。
- 保留冲突。
- 给出数据侧判断。
- 说明支持证据。
- 说明反驳证据。
- 说明失效条件。
- 给出 DataIntegrity 和 publish status。

它不需要理解当天新闻，也不需要为了新闻改结论。

### 3.2 它必须回答的问题

纯数据研报至少要回答：

1. NDX 当前主要矛盾是什么？
2. 利率和真实利率是否压制估值？
3. 信用利差是否确认风险偏好？
4. 估值和盈利质量是否能支撑当前价格？
5. 市场广度和集中度是否健康？
6. 趋势和技术状态是否只是短线拥挤？
7. 哪些数据互相确认？
8. 哪些数据互相冲突？
9. 哪些数据缺失会降低置信度？
10. 什么证据会让最终判断改变？

### 3.3 它不能越权

纯数据层必须继续遵守指标发言权：

- 技术指标不能证明估值便宜。
- 代理指标不能当作官方事实。
- 新闻 sidecar 不能成为 L1-L5 主证据。
- 当前网页不能伪装成历史回测当时可见信息。
- L5 价格趋势不能替代 L3 内部健康。
- 高估值不是自动看空，但高估值加高真实利率是核心冲突。

## 4. 第二层：新闻/事件简报

### 4.1 定位

新闻/事件简报是一份独立报告，不是纯数据研报的附属品。

它的任务是：

- 建立外部事件账本。
- 说明事件事实。
- 区分官方材料、公司披露、主流报道、市场叙事。
- 说明事件可能影响哪条金融链路。
- 标明来源、日期、限制和不确定性。

### 4.2 新闻来源分级

新闻层必须给每条材料标记来源等级。

| 来源等级 | 含义 | 能力边界 |
| --- | --- | --- |
| `official_fact` | Fed、BLS、BEA、财政部、SEC、Nasdaq、Invesco 等官方发布 | 可以作为事件事实，但仍需注意发布日期和数据日期 |
| `company_disclosure` | 公司 filings、财报、guidance、新闻稿 | 可以说明公司层面事实，不能直接推出整个 NDX 结论 |
| `primary_market_data_release` | 交易所、指数公司、基金公司、监管文件中的市场数据或规则变更 | 可以说明对象、权重、规则和产品变化 |
| `reliable_mainstream_report` | Reuters、Bloomberg、WSJ、FT、AP、CNBC 等主流报道 | 可作为有价值报道，但一般不能单独支撑强因果结论 |
| `sell_side_or_expert_view` | 投行、券商、专家、策略师观点 | 可作为观点材料，不能冒充事实 |
| `market_narrative` | 市场正在流行的解释、交易员叙事、社媒扩散主题 | 只能作为情绪和叙事线索 |
| `unverified_signal` | 未确认传闻、单点爆料、来源不清材料 | 默认不得进入综合强结论 |

### 4.3 新闻必须映射到金融链路

每条重要新闻不只要写“发生了什么”，还要写它可能影响哪条链路：

| 金融链路 | 例子 |
| --- | --- |
| `earnings_path` | 盈利预期、利润率、收入增长、资本开支 |
| `valuation_multiple` | 市场愿意给多少估值倍数 |
| `discount_rate` | 名义利率、真实利率、政策预期 |
| `risk_premium` | 投资者要求多少风险补偿 |
| `liquidity_condition` | 流动性、资金面、融资条件 |
| `credit_condition` | 信用利差、违约风险、融资压力 |
| `index_structure` | 权重、成分、集中度、等权与市值加权差异 |
| `market_breadth` | 上涨扩散、内部健康、广度修复或恶化 |
| `technical_flow` | 资金流、趋势、仓位拥挤、波动率结构 |

如果一条新闻说不清影响哪条链路，只能标为背景材料或弱线索。

### 4.4 新闻简报必须保留限制

每条重要新闻至少要说明：

- 事件日期。
- 发布日期。
- 来源。
- 事实部分。
- 解读部分。
- 可能影响的金融链路。
- 反证或限制。
- 是否需要后续数据确认。

这能防止“新闻看起来很重要，但其实只是噪音”。

## 5. 第三层：综合总报告

### 5.1 定位

综合总报告是三层架构的最终读者入口。

它读取：

- 第一层纯数据研报。
- 第二层新闻/事件简报。

然后做交叉质询：

- 数据侧最重要的判断是什么？
- 新闻侧最重要的事件是什么？
- 新闻是否解释了数据异常？
- 数据是否确认了新闻叙事？
- 数据是否反驳了新闻叙事？
- 两边哪里共振？
- 两边哪里冲突？
- 哪些解释最有说服力？
- 哪些解释只是可能？
- 哪些事情暂时解释不了？

### 5.2 综合解释等级

综合总报告必须为每个重要判断标注解释等级。

| 解释等级 | 含义 | 可用语气 |
| --- | --- | --- |
| `confirmed_fact` | 可确认事实 | “已经发生”“官方披露”“数据记录显示” |
| `data_supported_read` | 主要由正式数据支持 | “数据侧支持”“数据更倾向于” |
| `news_supported_read` | 主要由新闻或事件支持 | “事件侧提示”“新闻叙事指向” |
| `integrated_explanation` | 数据和新闻共同支持 | “两边互相确认”“较有说服力” |
| `plausible_hypothesis` | 合理但仍需验证 | “可能”“需要后续确认” |
| `weak_signal` | 弱线索 | “只能作为观察线索” |
| `not_explained` | 暂时解释不了 | “当前证据不足以解释” |

强结论只能来自 `confirmed_fact`、`data_supported_read` 或 `integrated_explanation`。

`news_supported_read` 不能单独变成强投资结论。

`plausible_hypothesis` 和 `weak_signal` 必须写清后续观察条件。

`not_explained` 必须被保留，不能被删除。

### 5.3 每个综合判断的标准结构

综合总报告中的每个重要判断都应采用同一结构：

| 字段 | 问题 |
| --- | --- |
| 判断对象 | 我们到底在判断 NDX、QQQ、NDXE、M7、半导体，还是某个宏观变量？ |
| 当前判断 | 现在倾向于什么？ |
| 数据支持 | 哪些正式数据支持？ |
| 新闻支持 | 哪些事件或报道支持？ |
| 反驳证据 | 哪些数据或新闻反对这个判断？ |
| 未解决张力 | 还有什么地方打架？ |
| 解释等级 | 属于哪一级解释？ |
| 置信度 | 高、中、低，为什么？ |
| 失效条件 | 什么发生后必须改判断？ |
| 后续观察 | 下一步看哪些指标或事件？ |

如果这些字段填不出来，就不应该写成强结论。

### 5.4 综合层的禁止事项

综合总报告不得：

- 用单条新闻证明复杂市场因果。
- 用新闻热度替代正式数据。
- 用 VIX 回落证明风险解除。
- 用某家公司财报证明整个 NDX 健康。
- 用 AI 叙事直接抵消估值压力。
- 用短线技术强势替代广度和估值判断。
- 把媒体观点写成事实。
- 把卖方观点写成系统结论。
- 把无法解释的冲突硬写成统一故事。

## 6. 来源标识与可选视觉辅助

### 6.1 必须做的是来源标识，不是颜色本身

综合总报告必须让读者一眼看出：

- 哪些是正式数据。
- 哪些是官方事实。
- 哪些是公司披露。
- 哪些是主流报道。
- 哪些是观点或叙事。
- 哪些是系统综合解释。
- 哪些是弱线索或未解释冲突。

这里的硬要求是“来源和强度必须标清楚”。

颜色、图标、标签样式只是表现层方案，不是架构核心。它们可以帮助普通读者降低误读，但必须经过实际页面验证：不能喧宾夺主，不能让读者误以为颜色本身代表真理，也不能因为颜色太多导致报告像看盘软件一样杂乱。

### 6.2 必需的来源标签

综合总报告应使用稳定的来源标签。标签本身比颜色更重要：

| 标签 | 含义 | 说明 |
| --- | --- | --- |
| `DATA` | 纯数据研报正式 evidence refs | 最高优先级的数据侧证据 |
| `OFFICIAL` | 官方事实 | 官方发布或监管材料 |
| `DISCLOSURE` | 公司披露、filings、财报 | 公司层面事实 |
| `MAINSTREAM` | 可靠主流报道 | 有价值报道，但通常不能单独支撑强因果 |
| `VIEW` | 投行、专家、卖方观点 | 观点材料，不是事实 |
| `NARRATIVE` | 市场叙事 | 情绪和叙事线索 |
| `HYPOTHESIS` | 系统假设 | 需要后续确认 |
| `CONFLICT` | 冲突或反证 | 必须显式保留 |
| `UNKNOWN` | 暂时无法解释 | 不能被强行解释掉 |

可选视觉方案：

- 可以用颜色辅助区分标签。
- 可以用图标辅助识别来源类型。
- 可以用边框、底色或小徽标提示冲突和未知项。

但这些都是 UI 选择，不是研究结论。任何颜色方案上线前，都要检查普通读者是否更容易理解，而不是更容易被颜色暗示。

### 6.3 读者入口上的展示规则

无论最终是否使用颜色，在综合总报告里：

- 支持证据旁边必须有来源标签。
- 反驳证据必须醒目标出。
- 弱线索不能和正式数据同色。
- 未解释冲突不能隐藏在脚注。
- `integrated_explanation` 不能只显示结论，必须能展开看到数据和新闻来源。

这能让普通读者不需要懂工程，也能看懂“这句话凭什么说”。

## 7. 关键矛盾与辩证法判断

### 7.1 主要矛盾

三层架构面对的主要矛盾是：

> 数据链的干净性 vs 现实解释的完整性。

只要纯数据，系统干净，但可能解释慢、解释窄。

只要新闻解释，系统灵活，但容易被叙事污染。

三层架构的价值，就是承认这两个目标都重要，并通过产物隔离解决矛盾。

### 7.2 次要矛盾

还存在几组次要矛盾：

| 矛盾 | 正确处理方式 |
| --- | --- |
| 快速解释 vs 审慎证据 | 可以快速提出假设，但必须标注为假设 |
| 可读性 vs 可审计性 | 总报告可以好读，但必须能展开证据 |
| 综合判断 vs 保留冲突 | 可以给主判断，但不能删除反证 |
| 新闻敏感性 vs 回测洁净 | 新闻可做当前解释，回测必须尊重 effective_date |
| 强结论需求 vs 证据不足 | 证据不足时必须降低置信度或拒绝强结论 |

### 7.3 架构结论

因此，正确路线不是：

- 禁止新闻。
- 也不是把新闻塞进 L1-L5。
- 更不是让总报告自由发挥。

正确路线是：

> 数据归数据，新闻归新闻，综合归综合；综合可以解释，但必须带着来源、等级、反证和失效条件解释。

## 8. 三份报告的读者入口

未来控制台或报告页应提供三个独立入口：

| 入口 | 给谁看 | 作用 |
| --- | --- | --- |
| 纯数据研报 | 想看正式数据链的人 | 看 L1-L5、Bridge、Risk、Final 的数据侧判断 |
| 新闻/事件简报 | 想看外部世界发生什么的人 | 看官方事件、公司披露、主流新闻和市场叙事 |
| 综合总报告 | 想看最终理解的人 | 看数据和新闻如何互相确认、冲突和解释 |

这三个入口都应该独立落盘。

它们可以互相引用，但不能互相覆盖。

综合总报告可以成为默认阅读入口，但必须能一键跳回：

- 原始纯数据研报。
- 原始新闻/事件简报。
- 每条关键判断的证据展开。

## 10. 工程接入路线

### 第一步：保留纯数据研报

不改现有主链的边界。

继续确保它能稳定生成：

- native brief。
- Prompt Inspector。
- Run Review。
- DataIntegrity。

验收标准：

- L1-L5 不接收新闻。
- Bridge / Thesis / Risk / Final 仍基于正式数据和 evidence refs。
- Prompt Inspector 未发现跨层污染。
- DataIntegrity / publish status 仍是发布闸门。

### 第二步：新增新闻/事件简报

新增独立流程，生成新闻/事件简报。

验收标准：

- 每条事件有来源、日期、事实、解读、限制。
- 官方事实、公司披露、主流报道、观点、叙事分级清楚。
- 新闻不改写纯数据研报。
- 新闻不进入 L1-L5 evidence refs。

### 第三步：新增综合总报告

新增 synthesis 流程，输入：

- `pure_data_report`
- `news_event_brief`

输出：

- `integrated_synthesis_report`

验收标准：

- 每个重要判断区分数据支持、新闻支持、综合解释。
- 每个强判断有反证和失效条件。
- 冲突和未解释项被保留。
- 颜色或标签能帮助普通读者识别来源等级。
- 综合报告不能绕过 DataIntegrity / publish status。

## 11. 发布闸门

综合总报告发布前必须过闸门。

如果出现以下情况，应降级、打标或拒绝发布：

| 问题 | 处理 |
| --- | --- |
| 纯数据研报 blocked / unpublishable | 综合总报告不能发布为正式结论 |
| 新闻来源混乱 | 只能发布为草稿或观察材料 |
| 强结论缺少数据证据 | 降级为假设 |
| 新闻和数据冲突但未说明 | 不通过 |
| 单条新闻支撑复杂因果 | 不通过 |
| 未解释冲突被删除 | 不通过 |
| 回测中使用了未来新闻 | 不通过 |

发布闸门的原则：

> 综合总报告不能比它的证据更自信。

## 12. 最终形态

理想的最终报告不是一篇顺滑长文，而是一份可以展开阅读的投研推理链：

- 顶部先给主判断。
- 然后说明判断对象。
- 再列主要矛盾。
- 再展示数据支持。
- 再展示新闻解释。
- 再展示冲突和反证。
- 再给失效条件和后续观察。
- 每一句关键判断都能追到来源。

普通读者能读懂。

专业读者能审计。

后续 agent 能继续追问。

这就是三层研报架构的目标：

> 既不让数据链被新闻污染，也不让报告对真实世界失明；既允许解释，又让解释服从证据、反证和失效条件。

## 13. 攻坚名称

后续重点工程可以命名为：

> 三层研报架构接入攻坚。

一句话任务：

> 保留现有纯数据研报，新建独立新闻/事件简报，再生成一个能交叉质询并谨慎解释的综合总报告；三份报告独立入口、独立落盘、互相引用但不互相污染；综合报告允许解释，但必须分级、可追溯、保留反证和未解释项。
