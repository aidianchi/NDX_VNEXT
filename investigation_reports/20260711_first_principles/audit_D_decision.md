# 审计 D：输出与决策转化层——第一性原理调查

调查范围：真实 run `output/analysis/vnext/20260709_233816`（当前唯一"完整"最新 vNext run）及其 HTML 判断书 `output/reports/vnext_brief_20260709_2338.html`；`src/agent_analysis/vnext_reporter.py`；`prompts/final_adjudicator.md`、`thesis_builder.md`；`src/state_ledger.py`、`src/agent_analysis/outcome_review.py`、`src/agent_analysis/run_review.py`；`PRODUCT.md`、`RUN_REVIEW_CHECKLIST.md`；`output/` 全库的历史 run 目录（约 30 个）。

所有结论标注【验证】（有 file:line 或 artifact 内容为证）或【推测】（基于间接迹象的合理推断）。

## ① 理想形态：单人投资者可靠 NDX 判断的输出与闭环应该长什么样

从决策科学角度（预测校准、Brier score、Superforecasting 式的"频繁小额下注+复盘"传统）出发，一个真正让判断变准的系统需要三层，缺一不可：

1. **判断产物层**：每次输出必须是可下注的（方向 + 仓位分桶 + 时间尺度 + 触发条件 + 置信度），而不是一篇论述。
2. **记录层**：每次判断必须被结构化记录为"押注"——押的是什么方向、多大确信度、什么条件下算错。
3. **校准层（最关键、最常缺失）**：系统必须**主动**在约定的未来时点回来核对"押注"，产出一个持续更新的、用户能看到的"过去 N 次判断，方向对了几次、哪类证据系统性偏乐观/偏悲观"面板，并且这个面板要能**反过来修正**下一次判断的措辞纪律或阈值（例如：如果历史上"战术仓边际改善"这个措辞后续总是不及预期，就该收紧这个用词的触发条件）。

更新节奏上，理想形态不需要每天出一份新判断书（NDX 不是日内交易品种，用户自己也说"每年 2-5 次关键决策"），但校准检查应该是**自动、周期性、不依赖人记得手动跑**的——比如每次 official run 之后自动挂一个"T+20/T+60/T+120 到期自动复核"的任务。

## ② 现状：这套系统实际做到了什么程度

### 判断产物层——做得比预期扎实

【验证】`final_adjudication.json` 和渲染出的 HTML 首屏（`output/reports/vnext_brief_20260709_2338.html`，`id="top"` / `id="decision"` 区块）包含：
- 方向性状态判断（"流动性紧缩下的高估值回调期…赔率中性偏弱"）
- 仓位分桶动作：核心仓/战术仓/等待资金三档，各自条件不同
- 具体触发价位：QQQ MA60=701、Donchian 下轨=686（`golden_pit_checklist.json:200`）
- 三个时间尺度（数日 / 1-3 个月 / 6-12 个月）各自的 view、action、失效条件
- confirmation_cost（等待确认的代价）、payoff_assessment（赔率判断）
- 五类"价格已经反映了什么"（信用/利率/估值/技术恐慌/流动性）分类判断
- 置信度标注（confidence: medium）、数据日期、指标覆盖率（30/42）

这不是一份"永远正确的模糊判断"。`prompts/final_adjudicator.md:14-20, 259-272` 和 `thesis_builder.md` 明确用大段"反模板约束"强制 LLM 点名当日主导矛盾、禁止套话，并且设了方向一致性硬规则（不能一边说赔率不利一边说高赔率）。

### 校准闭环层——设计存在，从未被真正启动过

这是本次调查最核心的发现，来自四条互相独立又互相印证的证据：

1. 【验证】`outcome_review_report.json` 在 20260709_233816（及查过的其余 ~20 个非测试 run）里全部是空壳：`"source": "not_run_for_live_or_non_backtest_context"`、`"windows": []`、`"claim_outcome_scores": []`。代码逻辑（`src/agent_analysis/outcome_review.py:387-412`）要求 `backtest_date` 存在才会跑，活当下（live）判断没有 `backtest_date`，所以每次真实使用都被跳过。
2. 【验证】`find output -name "claim_outcome_scores.json"` 在全仓库（约 30 个 run 目录）返回 **零个文件**——尽管这个产物的生成代码（`build_claim_outcome_scores`，`src/agent_analysis/outcome_review.py:329-366`）已经在 2026-07-07 的提交（`4b2163e Add claim outcome scoring and stage governance`）里写完并通过单元测试。代码存在，从未在任何一次真实运行里落过盘。
3. 【验证】`output/state_ledger/state_ledger.jsonl` 全库只有 **2 行**（对应 2 个 run，`20260707_163359` 和 `20260709_233816`），且两条都是 `"official": false`。而 `output/analysis/vnext/` 目录下有效 run 目录约 30 个，跨度从 4 月到 7 月。也就是说这条"跨 run 记录地基"实际上几乎没被用来记录任何东西。
4. 【验证，来自项目自己的待办文档】`NEXT_STEPS.md:26-27, 31` 三条 P1/P2 待办原文承认：
   - "claim outcome 打分器（T+20/60/120 verdict）只做过离线烟测，未经真实历史回测 run 验收真实判定质量"
   - "`config/user_decision_profile.json` 的买入/卖出 metric_predicates 阈值仍是草案，未经用户逐条确认"
   - "跨 run 展示层（跨 run diff、打分汇总）需要 ≥5 条真实台账才解锁"（目前是 0 条 official）

【推测，佐证性线索，来自不可信但可作旁证的 codex 体检文档】未追踪文件 `NDX_L1-L5_数据与推理链完整体检_20260710.md` 里也有一条独立同源记录："OPS-10 | P1 | Claim outcome 打分器只做离线烟测。| 未修"——与项目自己的 NEXT_STEPS.md 完全对得上，两份独立文档互相印证，可信度较高。

**结论：`state_ledger` / `outcome_review` 是"刚搭的空架子"，不是"真在用的闭环"。** 代码质量本身不差（有 no-backflow 声明、有单元测试、有 T+20/60/120 窗口设计），但截至审计时点，它在生产环境里的实际执行次数是 0（活判断）/ 2（离线测试性质的历史回测 smoke test，且这两次都发生在 claim-level 打分功能上线之前，只测了 market-level 窗口，不含 claim 级评分）。

### run_review.py——另一种"复盘"，但审的是流程不是结果

【验证】`src/agent_analysis/run_review.py:1-27` 的检查逻辑（`INTERNAL_READER_PHRASES`、`REQUIRED_PRICE_REFLECTION_CATEGORIES`、`HIGH_PAYOFF_TERMS` 等）全部是**文本模式匹配式的内部一致性检查**（比如检查有没有把内部审批话术泄露到读者结论里、payoff 措辞前后是否矛盾），不涉及"这次判断后来对不对"。这是另一层"过程质量"检查，和 outcome_review 的"结果质量"检查是两回事，容易被文档标题的"复盘"一词混为一谈。`RUN_REVIEW_CHECKLIST.md` 通篇的"五项复盘"（指标发言权、跨层冲突可追溯、客观性防火墙、L3 是否升级硬要求、brief 可读性）同样全部是过程检查，没有一项是"上次判断的方向后来被验证对不对"。

## ③ 关键发现（按严重度）

### 严重（系统性，直接影响"能不能变准"这个北极星问题）

**F1. 校准闭环在生产环境里从未真正跑起来过。** 证据见上文②节四条。这意味着无论 prompt 设计得多严谨（final_adjudicator.md 的反模板约束确实写得很仔细），**没有任何机制验证"严谨的措辞"是否对应"更准的判断"**。系统目前无法回答"过去半年这套系统给出的方向性判断，命中率大概是多少"这个最基本的问题——不是因为答案不好看，而是这个问题从未被问过。

**F2. 没有自动化的"回访"机制，且没有证据表明曾经手动回访过。** 即使未来某天有人想给 7 月 9 日这份判断打分，也需要人**记得**在几周后手动跑一条命令指向这个旧 run 目录。代码库里没有 cron/launchd/scheduler 迹象（【验证】搜索 WORK_LOG.md/NEXT_STEPS.md 未发现相关机制）。当前两条 state_ledger 记录都产生于 `run_pipeline` 结束时自动追加（`src/main.py:557-567`），是"写完就完"的一次性动作，不是"回来验证"的动作。

**F3. 驱动读者最关心的"距离买入/卖出纪律还差什么"板块的阈值，是未经用户确认的草案数字。** 【验证】`user_decision_profile.json:117` 明确标注 `"threshold_status": "draft_needs_user_confirmation"`，但 `golden_pit_checklist.json` 已经拿这些草案阈值（Forward PE ≤ 20、回撤 ≥ 15%、VIX 分位 ≥ 0.7 等）去做真实的 `met/not_met` 判定，并直接渲染进 HTML 首屏最显眼的板块（`vnext_reporter.py:1910` `距离买入/卖出纪律还差什么`）。用户如果没意识到这些数字仍是草案，可能把"证据不足 / 未满足"当成对自己个人纪律的确定性判断，而实际上连纪律本身的门槛都还没被用户拍板过。

### 高（影响单次报告的可读性/可信度，但不动摇整体架构）

**F4. UI 把"已验证的风险 claim"和"真正的买卖纪律触发条件"用相同视觉语言混排，容易误导快速通读的用户。** 【验证】`vnext_reporter.py:1863-1898`：渲染"距离买入/卖出纪律还差什么"清单时，代码**不区分** `discipline_side` 字段（`claim` vs `buy` vs `sell`）。结果是本轮 checklist 的前三条（`discipline_side: "claim"`，内容是"NDX 估值处于历史偏高区间…风险补偿偏低"这种利空描述）和后四条真正的买卖纪律预测（`discipline_side: "buy"/"sell"`）用了完全相同的绿色"已满足"pill 样式并列展示（`output/reports/vnext_brief_20260709_2338.html` 中 `距离买入/卖出纪律还差什么` 小节，前三条均为绿色"已满足"）。对于"通读、不细看审计区"的目标用户画像来说，三个绿色"已满足"排在清单最前面，视觉上很容易被读成"利好信号已确认"，实际含义却是"这条利空风险描述本身被验证为准确"——这是两种性质完全相反的信息，用了同一种视觉语言。

**F5. golden_pit_checklist 的证据/反证是整批复制，不是逐条精确匹配（团队自己已发现，未修）。** 【验证】`NEXT_STEPS.md` P2 行原文："`golden_pit_checklist.json` 生成阶段把全量 evidence_refs 和反证整包写进每个条目，条目间无区分，报告层只能做共用块兜底"。实际 JSON（`golden_pit_checklist.json` 全部 7 个条目）证实：几乎所有条目的 `falsification_conditions` 列表高度重复雷同，报告层只能靠一个"共用块"折叠展示（`vnext_reporter.py:1899-1908`）。这削弱了"什么情况下我错了"这个可证伪性问题在**条目粒度**上的精确度——目前更像是"整批判断共享一套大而化之的反证清单"，而不是"每条具体纪律各自对应精确的失效触发器"。

**F6. 跨 run 对比被硬编码为"暂缓"，即便已经积累了几十次运行。** 【验证】`src/agent_analysis/orchestrator.py:2584` 直接写死 `"status": "deferred_until_run_quality_stable"`；`golden_pit_checklist.json` 全部 7 个条目的 `changed_since_last_run.status` 都是这个值。这是"判断是否在漂移/稳定"这一校准最基本信号的直接来源，目前被主动关闭，而不是因为技术做不到——`state_ledger.py` 已经把跨 run 需要的确定性状态变量都提取好了，只是没有拿来做比较。

### 中（值得改进，但不构成结构性风险）

**F7. "谁在竞争解释权"（反方假说）机制结构精致，但对抗性内容偏薄。** 【验证】本轮 `hypothesis_competition.json` / `counter_thesis.json` 渲染出的"反方最强论证"实质内容是"现有证据更像是未解决张力和证据缺口，不足以支持…"——本质上是在说"证据不够"，而不是提出一个真正相反方向的替代判断（例如"这其实已经是买点，因为…"）。反方"自己也承认无法确立的点"清单（利率数据缺失、盈利修正数据薄、信用利差缺失）和主线的"它解释不了"清单几乎重合。这套机制目前更像是在展示"我们知道哪里证据不够"，而不是真正压力测试主线判断的方向性。

**F8. 我读到的这份"最新完整" run 官方自己承认已经过时，不能作为验收基准。** 【验证】`RUN_REVIEW_CHECKLIST.md:12` 原文："最新完整 vNext 仍是 `20260709_233816`，它早于本轮 L3 稀疏日/补洞、薄元信息、History of Market 低样本分位和 prompt 修复，不能当作这些修改的 fresh 验收。" 本审计依据的样本，是当前唯一可用的"完整链路"真实产物，但项目方自己已经把它标记为"待重跑验证"。这不改变本审计发现的架构性问题（校准闭环缺失是代码逻辑层面的问题，与具体某次 run 无关），但意味着本报告对**单份判断书文字质量**的具体评价，可能已被后续未验收的修复部分覆盖。

**F9. 审计机制能守住"证据格式"，但守不住"数字本身对不对"——有真实案例。** 【验证】`WORK_LOG.md:102` 记录 2026-07-08 的一次人工"第一性原理排查"发现：L1 净流动性数值被 L1 agent 写错 10 倍（"5815.95 亿"实为 5815.95 十亿）。这个错误在被发现之前，已经通过了 schema guard、evidence_refs 追溯、objective firewall 等全部审计闸门——因为这些闸门检查的是"有没有引用来源""格式对不对""风险有没有保留"，不检查"这个具体数字对不对"。这个案例是"可审计"和"可靠"之间真实鸿沟的具体证明，不是理论担忧。（本条影响未在 20260709_233816 run 中被直接复核到，L1 叙事此轮显示单位为"B"，看起来已避开了这个具体表现形式，但案例本身足以说明审计机制的盲区类型。）

**F10. 失效条件是定性的、非机械可验证的，这是刻意的设计选择，但也让"可证伪性"停留在纸面。** 【验证】`final_adjudicator.md:14-17` 和 `thesis_builder.md:295` 明确禁止编造历史胜率、点位阈值、概率数字——这是认知诚实的合理边界，不是缺陷。但代价是失效条件多为"盈利预期结构性下修""信用继续加速恶化"这类需要主观判断的措辞，而不是像 `golden_pit_checklist` 里 `metric_predicates` 那样的硬数字触发器（后者其实做得很好，例如 `credit.hyg_percentile_1y <= 5`，本轮实际值 8.7，`met: false`，是真正机械可评估的）。问题是：这类硬触发器目前只用在 `user_decision_profile` 的 buy/sell discipline 上，没有延伸到 Final 判断本身的 `invalidation_conditions`（那部分仍是自然语言）。而且——回到 F1/F2——即便条件本身是可证伪的，也没有机制真的回去检查它有没有被触发。可证伪性目前是"结构上具备潜力"，不是"已经被验证在起作用"。

## ④ 总判断：保留 / **改良** / 推倒重来

**改良，不是推倒重来。**

理由：

- 判断产物层（final_adjudication → thesis_builder → golden_pit_checklist → HTML 渲染）这条链路的设计质量出乎意料地扎实：有方向、有仓位分桶、有时间尺度、有具体触发价位、有反模板措辞纪律、有五类价格反映拆解。这不是"证据齐全但判断平庸"的空转系统——从这份样本判断书看，它确实点名了具体的主导矛盾（宏观流动性 vs 高估值），没有回避"利率数据全部缺失"这个真实的信息缺口，措辞上也没有滑向套话。**推倒重来会浪费这部分已经做对的工作。**
- 真正的缺口高度集中在**一个具体机制**：校准闭环。而这个机制的代码骨架（contracts、no-backflow 声明、T+20/60/120 窗口设计、claim-level 打分器、聚合脚本）已经写完，只是从未在生产环境里真正执行过一次。这是**执行/运营层面的缺口，不是架构层面需要重新设计的缺口**——换句话说，这更接近"量变"（把已经造好但没通电的机器插上电、按周期运行、把结果做成用户能看到的样子），而不是"质变"（推翻现有 schema 重新设计）。
- 项目自己在 2026-07-07 也已经把状态从"施工期"切换为"架构冻结、进入使用期"（`NEXT_STEPS.md:11`），这与本次审计的结论方向一致：现在需要的不是造更多新阶段/新 artifact，而是让已经造好的校准机制真正跑起来、并被用户看到。

需要"质变"的部分只有一处：**目前完全没有设计"读者可见的历史准确率反馈"这个呈现层**——即便 outcome_review 和 claim scoring 真的跑起来了，现在也没有任何 UI/artifact 形态是"打开报告，看到过去 N 次判断的命中率趋势"。这部分需要新设计，不是简单调用已有代码就能补上。

## ⑤ 给总报告的 3-5 条核心启示

1. **"可审计"不等于"可靠"，这套系统提供了前者，后者目前无法验证——不是因为判断差，而是因为验证机制从未被启动过。** 校准闭环（state_ledger + outcome_review + claim_outcome_scores）代码已就绪但生产环境执行次数为 0（活判断）。这是当前最大的单点缺口，优先级应高于任何报告呈现层的打磨。

2. **判断本身的形态并不差，不需要因为"担心系统只会说模糊话"而重新设计判断生成链路。** 本轮样本证明系统能给出具体仓位分桶、时间尺度、触发价位和主导矛盾，问题不出在"敢不敢下结论"，而出在"下完结论之后没人回来对账"。

3. **在完善校准闭环之前，"距离买入/卖出纪律还差什么"这个最贴近决策的板块存在两个会误导用户的具体问题**：(a) 驱动它的数字阈值仍是用户未确认的草案（`user_decision_profile.json`），(b) UI 没有区分"利空风险已被验证"和"买卖条件已满足"，两者用了同一个绿色"已满足"标签并列展示。这两点修复成本不高（一个是要求用户过一遍确认流程，一个是渲染层加一个 discipline_side 分组/配色区分），但对决策可用性的影响立竿见影，建议列为高优先级。

4. **"可审计推理链"这个北极星本身没有问题，但需要明确它是必要不充分条件。** 净流动性 10 倍单位错误的真实案例证明：全套审计闸门（schema guard、evidence_refs、objective firewall）可以在数字本身错误的情况下全部通过。建议把"事后结果校准"正式提升为与"过程审计"同等地位的第二支柱，而不是让它停留在"使用期遗留事项"的待办列表里。

5. **量变 vs 质变的判断：判断生成和校准骨架是量变（需要执行、不需要重新设计），读者可见的"历史准确率反馈"呈现层是质变（目前不存在，需要新设计）。** 建议优先级：先让 outcome_review 在 official run 上真正跑起来并积累 ≥5 条记录（项目自己设的解锁门槛），再设计一个用户会主动去看的"过去判断复盘"入口——否则即便数据攒够了，用户也不会知道去哪看、看到了会不会改变行为。
