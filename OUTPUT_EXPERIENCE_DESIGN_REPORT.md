# vNext 输出体验深度报告

对象：`output/reports/vnext_research_ui_brief_20260502.html`  
阶段：实施前设计报告；2026-05-03 已据此完成第一轮结构改造，但该改造版不是终版

## 2026-05-03 用户验收补记

用户已查看第一轮改造版 `output/reports/vnext_research_ui_brief_20260502.html`，反馈非常明确：

- 当前版本只能作为中间版本保留，不是终版，距离理想输出体验仍差得远。
- 当前审美方向不满意，尤其主视觉配色不应作为后续默认方向。
- 五层底稿区域的点击、展开、跳转和动效反馈不够顺畅，用户感知为“似乎无法跳转，动画有问题”。

因此，本报告中的原则仍然有效，但第一轮落地结果不能视为设计方向已经被验证。后续只保留方向性判断：审美需要重新定调；交互需要继续打磨；图表、数据和报告的打开方式需要更自然、更低门槛。

## 1. 一句话结论

vNext 最终给用户看的“答卷”不应该只是一个更漂亮的 HTML，也不应该只是一个数据仪表盘；它应该是一个可连续阅读的投研长卷，同时带有一键展开的证据地图、图表抽屉、数据来源说明和审计路径。

更准确地说，它应该有三层：

1. **读得懂的主报告**：像一篇高质量投资备忘录，先告诉用户结论、理由、风险和什么会改变判断。
2. **打得开的证据层**：任何结论旁边都能打开相关指标、图表、来源、覆盖率、更新时间和反证条件。
3. **可审计的底稿层**：研究员或开发者可以继续追到 L1-L5、Bridge、Critic、Risk、Final 和原生 artifact。

这三层必须同时存在，但默认只让普通用户看到第一层和必要的第二层。审计能力要随手可达，而不是把原始 JSON 堆在页面末尾让人硬啃。

## 2. 当前 brief 已经做对了什么

最新 brief 页面已经有很重要的基础，不是从零开始：

- 它直接消费 vNext v2 artifacts，而不是依赖 legacy adapter 拼主要叙事。
- 它保留了 Final、风险、五层底稿、Bridge 冲突、治理和审计。
- 它提供了 evidence ref 点击跳转，证明“结论能追到指标卡”这条路已经打通。
- 它能展示 L4 估值源、Trendonify 403、WorldPERatio、Damodaran、覆盖率和口径边界，说明数据发言权已经进入 UI。
- 它的默认 brief 模板已经接近“连续阅读”，比 cockpit / atlas / workbench 更适合作为用户入口。

这意味着下一步不是推翻，而是把它从“artifact 浏览器”升级为“用户真的愿意读完、能读懂、能追问、愿意信任的投研答卷”。

## 3. 当前 brief 的主要阅读卡点

### 3.1 阅读顺序仍像内部底稿

当前 brief 顺序是：

```text
裁决 -> 五层底稿 -> 证据链 -> 冲突 -> 治理 -> 审计
```

这个顺序保证底稿完整，但对普通用户并不顺。用户刚看到最终结论后，最自然的问题不是“L1-L5 每层完整底稿是什么”，而是：

1. 为什么是这个结论？
2. 哪些证据最关键？
3. 最大风险是什么？
4. 哪些证据互相冲突？
5. 什么情况会推翻这份判断？
6. 如果我想深挖，原始指标在哪里？

所以主阅读顺序应该更像：

```text
结论摘要 -> 核心理由 -> 最大风险 -> 冲突地图 -> 关键图表/数据 -> 五层展开 -> 治理审计
```

五层底稿很重要，但不应该在主阅读流中过早占据大量篇幅。

### 3.2 证据跳转有合同问题

页面里有些 evidence ref 是干净的，例如：

```text
L1.get_10y_real_rate
```

但有些 ref 带了读数和解释，例如：

```text
L1.get_10y_real_rate: 1.94% 高分位
```

当前点击逻辑会用整个字符串生成目标 id。带冒号、空格、中文解释的 ref 很难命中真实指标卡。这个问题会破坏用户最重要的信任动作：点一下证据，看见原始依据。

后续应该把 evidence ref 分成两层：

- `ref_id`：稳定机器 id，例如 `L1.get_10y_real_rate`。
- `label`：给人看的说明，例如 `10Y 实际利率 1.94%，处于高分位`。

用户看到的是 label，系统跳转使用 ref_id。

### 3.3 页面语言还暴露了太多后台结构

当前页面有不少英文或系统字段：

- `Final Adjudication`
- `Must-Preserve Risks`
- `Evidence Graph`
- `Bridge & Conflict Map`
- `Layer Workbench`
- `approved_with_reservations`
- `normalized_state`
- `function_id`

这些词对系统开发有用，但对普通读者会制造距离感。最终报告可以保留 L1-L5 这类核心架构名，但第一次出现要翻译，并把后台字段换成读者语言。

例如：

- `Final Adjudication` -> `最终判断`
- `Must-Preserve Risks` -> `不能淡化的风险`
- `Evidence Graph` -> `结论依据`
- `Bridge & Conflict Map` -> `证据之间的冲突与共振`
- `approved_with_reservations` -> `通过，但保留风险`

### 3.4 信息密度很高，但缺少渐进展开

当前页面把很多内容一次性摊开：层级综合、内部冲突、hooks、质量自检、指标卡、数据质量、原始 JSON。它对审计友好，但阅读压力大。

顶级报告应该遵循：

> 默认给结论和关键证据，用户想深挖时再展开细节。

具体应当分三档：

1. **一句话档**：结论、置信度、四个最大风险、最关键冲突。
2. **正常阅读档**：每个判断对应 2-4 条证据和 1 条反证。
3. **审计档**：指标卡、来源、覆盖率、fallback、原始 artifact。

这三档不应该互相替代。

### 3.5 图表和数据还没有成为一等公民

当前报告主要是文字和卡片，数字被写在段落里。对于投研报告，这还不够。

用户应该能随手打开：

- 当前值。
- 历史分位。
- 近期走势小图。
- 来源和更新时间。
- 覆盖率。
- fallback 链。
- 这个指标能证明什么。
- 这个指标不能证明什么。
- 哪个反向证据会推翻它。

图表不应为了装饰存在。每个图表都必须回答一个具体问题：

- 利率压力是在升还是降？
- 估值贵在哪里？
- 趋势强但是否过热？
- 广度健康还是头部集中？
- 风险偏好是在支撑还是在过度乐观？

### 3.6 审美已经有方向，但还没有到“登峰造极”

当前 brief 的气质是温暖、纸感、研究报告感，这是对的。但它还有几个问题：

- 色彩偏米色、棕色、暖绿，容易形成单一复古纸面感。
- 大量圆角卡片让页面更像现代 dashboard，而不是顶级研究长卷。
- Hero 字号巨大，但中文负字距和超大标题会降低精致度。
- 字体 fallback 偏普通，尚未形成“金融研究院级别”的独特识别。
- 卡片层级多，视觉上容易每块都重要，反而削弱真正重要的风险和冲突。
- 缺少足够精密的数据视觉语言，例如细线、刻度、分位尺、状态轴、证据链路径。

下一版应该追求的不是“更炫”，而是“更有判断力、更有秩序、更高级”。

## 4. 顶级用户答卷的核心原则

### 原则 1：先让用户知道结论，但不能让结论压过风险

第一屏必须回答：

- 现在对 NDX 的判断是什么？
- 置信度是多少？
- 这份判断最依赖哪三条证据？
- 哪些风险绝对不能淡化？
- 什么情况会让判断失效？

用户不应该滚动五分钟才知道重点。

但第一屏也不能变成“标题党结论”。vNext 的核心资产是冲突，所以第一屏必须同时显示：

```text
最终判断 + 核心理由 + 最大未解决张力
```

### 原则 2：报告是主线，证据是侧线，审计是底线

主线负责顺滑阅读，侧线负责随手展开，底线负责追责和复核。

三者关系应该是：

```text
主报告：我现在怎么看
证据层：我为什么这样看
审计层：这套判断是否干净、可复现、有没有越权
```

不能把三者混在一个长页面里平铺，否则普通读者会累，专业读者也会找不到重点。

### 原则 3：每一个判断都要有“证据、反证、触发器”

顶级投研报告不只是“我看多/看空”。它必须告诉用户：

- 支撑证据是什么。
- 反向证据是什么。
- 现在冲突在哪里。
- 未来什么信号会改变判断。
- 如果错了，最可能错在哪里。

vNext 的界面应该把这个结构固定下来，让每个重要判断都长成同一种可审计形态：

```text
判断
  -> 支撑证据
  -> 反向证据
  -> 未解决张力
  -> 观察触发器
  -> 证据来源
```

### 原则 4：冲突不是附录，冲突是主角之一

传统报告喜欢把冲突写成风险提示，放在后面。vNext 不能这样。

vNext 的优势是能显式保留：

- 高实际利率 vs 高估值。
- 趋势强 vs 超买。
- 广度健康 vs 集中度极端。
- 风险偏好强 vs 情绪逆转风险。
- 盈利增长预期 vs 现金收益率低。

所以页面应该有一个非常醒目的“冲突地图”。它不是风险附录，而是读者理解结论强弱的核心界面。

### 原则 5：图表必须服务于理解，不服务于装饰

任何图表都应该满足至少一个目的：

- 看方向。
- 看位置。
- 看极端程度。
- 看分歧。
- 看来源可靠性。
- 看是否有数据缺口。

建议优先建立以下图表语言：

- **分位尺**：显示当前值处于历史什么位置。
- **走势小图**：显示近 20 / 60 / 252 日变化。
- **差距条**：显示 FCF Yield 与 10Y Treasury 的差距。
- **冲突轴**：左侧证据、右侧反证，中间是当前判断。
- **传导路径图**：L1 -> L4、L2 -> L4、L1 -> L5 等机制。
- **覆盖率徽章**：明确数据覆盖多少、缺了什么。

### 原则 6：普通用户不需要懂代码，也能追到证据

用户不应该看见 `function_id` 才知道这是指标。

更好的呈现是：

```text
10Y 实际利率
1.94%，处于高分位
来源：FRED
用途：判断折现率压力
不能证明：不能单独证明股市必跌
点击展开：完整指标卡
```

机器 id 可以保留在高级模式、复制链接、审计面板里，但不应成为默认阅读语言。

### 原则 7：审美要像“顶级研究院”，不是“好看的 SaaS”

这个项目不是普通运营后台，也不是营销页。它的审美应该接近：

> 高级投研长卷 + 战略地图 + 审计档案馆。

关键词应该是：

- 克制。
- 准确。
- 有层级。
- 有纸面质感但不复古过头。
- 有数据密度但不压迫。
- 有冲突张力但不戏剧化。
- 像可信机构的正式答卷，而不是模板化 dashboard。

建议视觉方向：

```text
Quiet Institutional Editorial
安静、机构级、编辑化、可审计
```

它可以高级，但不能炫耀；可以美，但不能抢走判断本身。

## 5. 推荐的信息架构

### 5.1 默认入口：一份“投研长卷”

默认载体仍建议先保持 self-contained HTML，因为当前信息架构尚未完全稳定。HTML 仍然最适合快速迭代、离线保存、审计和分享。

但它不能只是现在的长页面。建议下一版 brief 采用以下结构：

```text
0. 封面摘要
1. 最终判断
2. 为什么是这个判断
3. 不能淡化的风险
4. 关键冲突地图
5. 会改变判断的触发器
6. 五层证据摘要
7. 指标与图表抽屉
8. 治理与审计
```

### 5.2 第一屏应该长什么样

第一屏不需要巨大标题占满空间。它应该像一页高质量投资备忘录的封面：

- 左侧：最终判断。
- 下方：一句话解释。
- 右侧：置信度、数据日期、指标覆盖、审批状态。
- 底部：四个不能淡化的风险。
- 一条细的状态轴：支持 / 反对 / 未解决。

第一屏应该让用户在 20 秒内知道：

```text
这份报告不是在喊方向，而是在说明风险收益比不利。
主要问题不是趋势弱，而是估值、流动性、集中度和超买之间的张力。
```

### 5.3 主报告应该像“可折叠的长文”

主报告不应只堆卡片。它应该有清晰段落：

- 当前判断。
- 支撑这个判断的三条主线。
- 反对这个判断的证据。
- 为什么最终仍然选择中性偏谨慎。
- 未来观察点。

每一段旁边都有证据芯片，点击后在右侧或底部打开证据抽屉，不打断阅读位置。

### 5.4 五层底稿应该从“默认展开”变成“按需展开”

L1-L5 是系统骨架，但普通用户不应默认被五层全文淹没。

建议改成：

- 主文中只显示每层一句结论。
- 每层有一个小型状态卡：方向、置信度、核心指标、最大内部冲突。
- 点击后展开完整 Layer：综合、内部冲突、hooks、质量自检、指标卡。

这样保留上下文隔离和审计能力，同时让主阅读流更轻。

### 5.5 证据地图应该变成可视化核心

证据链目前是列表。下一版应升级成“证据地图”：

- 每条证据链是一条横向路径。
- 每个节点是指标或层级。
- 节点颜色代表支持、反对、警告、缺口。
- 节点点击打开指标详情。
- 边代表传导机制，例如 `实际利率 -> 折现率 -> 估值压力`。

这会把 vNext 和普通报告明显区分开。

### 5.6 风险和触发器应该有单独区域

当前风险写得完整，但阅读上还像列表。建议改成：

```text
风险名称
严重度
为什么重要
当前证据
触发条件
会推翻风险的反证
观察指标
```

例如：

```text
估值压缩风险
严重度：高
当前证据：PE 33.83、FCF Yield 1.55%、10Y 4.4%、实际利率 1.94%
触发条件：实际利率继续上行、盈利增速下修、净流动性持续转负
反证：实际利率回落、盈利显著上修、风险偏好和资金流继续确认
```

## 6. 推荐的交互原则

### 6.1 一键展开证据，但不丢失阅读位置

证据点击不应该直接把用户卷到页面深处后找不回来。更好的方式是：

- 桌面端：右侧证据抽屉。
- 移动端：底部抽屉。
- 抽屉里显示指标卡、图表、来源、数据质量和原始 ref。
- 抽屉内有“跳到完整底稿”按钮。

这样普通用户可以轻读，专业用户可以深挖。

### 6.2 提供阅读模式和审计模式

同一份报告应有两个模式：

- **阅读模式**：默认，隐藏技术字段，突出结论、风险、图表、解释。
- **审计模式**：显示 function_id、artifact、schema、raw JSON、token usage、来源链。

这比生成多个模板更自然。cockpit / atlas / workbench 可以保留为内部视图，但用户入口最好是一个报告内切换。

### 6.3 导航不是目录，而是阅读仪表

导航应该告诉用户当前位置和阅读完成度，而不只是锚点链接。

建议包含：

- 当前章节。
- 结论、证据、冲突、风险、底稿、审计。
- 关键风险数。
- 断裂证据数。
- 数据缺口数。

### 6.4 所有证据 ref 必须可复制、可定位、可恢复

每个证据节点应支持：

- 点击打开。
- 复制 ref。
- 复制该证据的短说明。
- 生成锚点链接。
- 回到刚才的阅读段落。

这是投研系统“可审计”的用户体验基础。

### 6.5 移动端不能只是压成一列

当前 responsive 主要是栅格变一列。顶级体验需要移动端单独设计：

- 第一屏减少超大标题。
- 风险卡横向滑动或折叠。
- 证据抽屉从底部弹出。
- 导航变为底部章节条。
- 指标图表使用迷你图，不把大表格塞进窄屏。

## 7. 推荐的审美方向

### 7.1 视觉定位

建议方向：

```text
Quiet Institutional Editorial
安静的机构级编辑化报告
```

它不是黑金豪华风，不是科技蓝紫风，也不是复古羊皮纸风。它应该像一份来自顶级研究机构的正式答卷：

- 页面留白克制。
- 字体有中文阅读质感。
- 数据用细密线条和精准刻度。
- 风险色不刺眼，但一眼可见。
- 图表像研究工具，不像装饰元素。

### 7.2 色彩

建议避免继续让米色、棕色、绿色占据全部气质。可以保留纸感底色，但引入更清晰的数据色系：

- 背景：温白、冷灰白、轻纸色。
- 主文字：近黑但不纯黑。
- 结构线：低透明灰。
- 支持证据：深绿或墨绿。
- 风险证据：克制红。
- 中性/观察：琥珀。
- 数据/链接：深蓝灰。
- 审计/原始数据：石墨灰。

关键是减少大面积渐变和大面积暖色，让内容更像正式研究档案。

### 7.3 字体

中文长文需要优先保证阅读舒适。建议：

- 标题用有气质的中文宋/明体或屏显宋体。
- 正文用高可读中文黑体或宋体，不要过度压缩。
- 数据和 ref 用等宽字体，但只在数据位置使用。
- 中文标题不要使用过强负字距。

字体的目标不是“特别”，而是让报告有机构级识别，同时长时间阅读不累。

### 7.4 卡片和布局

当前大圆角卡片太多，容易让页面像 SaaS dashboard。下一版应减少“卡片套卡片”，改成：

- 主文是连续纸面。
- 关键风险是清晰块状提示。
- 指标卡是可展开组件。
- 审计内容使用抽屉或附录。

边框、分割线、编号、刻度和小图表应承担更多层级表达。

### 7.5 动效

动效要少，但要准：

- 打开证据抽屉时平滑进入。
- 点击证据时高亮对应指标。
- 图表 hover 显示读数。
- 切换阅读/审计模式时保留当前位置。
- 展开层级时避免页面大跳动。

不要使用炫技动效。投研报告的动效应该服务于定位和理解。

## 8. 推荐载体判断：HTML、正式 viewer，还是别的？

### 短期：继续 self-contained HTML

原因：

- 当前信息架构仍在验证。
- HTML 容易离线保存和分享。
- 方便直接从 run artifacts 生成。
- 不需要引入正式前端工程成本。

但下一版 HTML 应该从“静态长页”升级为“自包含交互报告”。

### 中期：准备正式 viewer

当以下条件满足后，再考虑正式 viewer：

- brief 信息架构稳定。
- 证据 ref 合同稳定。
- 图表组件稳定。
- 阅读模式和审计模式被验证有效。
- 真实用户确认默认阅读顺序合理。

正式 viewer 的意义不是“更现代”，而是支持：

- 多 run 对比。
- 历史报告归档。
- 指标趋势跨 run 追踪。
- 证据搜索。
- 交互式图表。
- 用户注释。

### 长期：多载体输出

最终可以形成三种输出：

- **交互 HTML / viewer**：主产品。
- **PDF 导出版**：给归档、分享和正式汇报。
- **JSON artifact 包**：给审计、自动化和二次分析。

但现在不要急于 PDF 优先。vNext 的核心优势是可展开和可审计，PDF 只能作为导出，不应成为主载体。

## 9. 下一步实施前应达成的设计决议

在动手改 UI 前，建议先确认以下决议：

1. 默认入口是否仍叫 `brief`，还是升级命名为 `report`。
2. 主阅读顺序是否改为：结论 -> 依据 -> 风险 -> 冲突 -> 触发器 -> 五层 -> 审计。
3. evidence ref 是否立刻改成 `ref_id + label` 双层结构。
4. 五层底稿是否默认折叠，只在主文显示摘要。
5. 是否加入证据抽屉，而不是只做页面跳转。
6. 图表第一批先做哪些：分位尺、走势小图、估值差距条、冲突轴、传导路径。
7. 审美方向是否确认走“安静的机构级编辑化报告”，而不是继续大圆角 dashboard。

## 10. 建议实施路线

### 第一阶段：信息架构重排

目标：先让报告顺。

- 调整 brief 主章节顺序。
- 第一屏重新组织为结论、置信度、数据日期、指标覆盖和最大风险。
- 五层底稿默认摘要化。
- 风险和冲突前置。
- 技术字段中文化。

完成标准：

- 普通读者 3 分钟内能说清结论和最大风险。
- 专业读者能在 1 次点击内看到关键证据。

### 第二阶段：证据合同修复

目标：所有证据点击都稳定。

- evidence ref 拆成 `ref_id` 和 `label`。
- 所有芯片用 ref_id 定位。
- 带解释文字的证据不再破坏跳转。
- 点击后打开证据抽屉，并提供完整底稿跳转。

完成标准：

- 页面内所有证据芯片点击都能命中。
- 没有无效 ref 静默失败。

### 第三阶段：图表与数据组件

目标：让数据成为一等公民。

- 指标卡加入分位尺。
- 关键指标加入小走势。
- L4 加估值差距条。
- Bridge 加冲突轴和传导路径图。
- 数据质量以来源、覆盖率、更新时间、fallback 展示。

完成标准：

- 用户不用读完整段文字，也能看懂核心数据位置和风险方向。

### 第四阶段：审美系统升级

目标：从“漂亮原型”升级为“顶级报告”。

- 建立完整色彩变量。
- 调整字体系统。
- 减少大圆角卡片堆叠。
- 改善移动端。
- 加入克制动效。
- 清理过多后台英文。

完成标准：

- 页面看起来像正式投研答卷，不像模板化 dashboard。
- 长文阅读不累，数据展开不乱。

### 第五阶段：验证与真实 run 回归

目标：证明它不是静态美化。

- 用最新 run 生成报告。
- 检查所有 ref 跳转。
- 检查移动端文字不溢出。
- 检查风险、冲突、反证没有被隐藏。
- 检查 raw artifact 仍可打开。

完成标准：

- 审美提升不牺牲可审计性。
- 阅读顺滑不抹平冲突。

---

## 11. 刻意不建议现在做什么

暂时不建议：

- 直接上正式 React/Vue 前端。
- 把 cockpit 做成默认入口。
- 用大量动画制造“高级感”。
- 为了页面顺滑删掉冲突、反证和数据缺口。
- 把 raw JSON 当成普通用户的审计体验。
- 继续美化 legacy HTML。
- 用装饰性图表替代真正的数据解释。

现在最重要的是先把“答卷形态”定准。

---

## 12. 最终建议

我建议下一步把 brief 升级为一个真正的 `report` 形态：

```text
主线是可连续阅读的投研长卷；
侧线是随手打开的证据、图表和数据来源；
底线是完整保留的 vNext 审计链。
```

审美上走“安静的机构级编辑化报告”，而不是普通 dashboard。交互上优先做证据抽屉、稳定 ref、冲突地图和指标图表。工程上继续保持 self-contained HTML，等信息架构和组件语言稳定后，再决定是否做正式 viewer。

这条路线最符合 vNext 的核心原则：

> Context-first, role-second.  
> 冲突是资产。  
> 原生 v2 artifact 是主产品。  
> 用户看到的不是漂亮包装，而是一条可信、可展开、可审计的投研推理链。
