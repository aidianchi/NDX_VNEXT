从第一性原理推回测原则
1. 回测的本质
回测 = 用历史数据评估一个决策系统在过去某些时点的表现，借此推断它对未来的有效性。

这条定义里有一个关键假设：回测时点上系统的"信息边界"必须等同于未来真实运行时的"信息边界"。否则你测的就不是同一个系统。

形式化讲，对每个回测时点 t：

系统可见信息 I_t = 在 t 及之前可被观测到的全部数据
系统决策 D_t = f(I_t) ，其中 f 是被测试的系统
决策后果 P_{t+k} 在 t+k 时点才知晓
评估 = 用 (D_t, P_{t+k}) 对的多样本统计
要让这个测试有外推价值，必要条件就 4 个：

(A) 信息不穿越时间：f(I_t) 真的只用了 I_t，没用任何 t 之后的信息。

(B) 决策可重现：相同 (t, I_t) 必须能重新产出相同 D_t（有随机性时 seed 可控）。

(C) 评估独立性：P_{t+k} 不能反过来影响 D_t 的生成（不能用结果选样本）。

(D) 样本无幸存者偏差：选用的 t 集合不能因为后来发生的事被选择性排除。

C. 系统根本没意识到的盲区（5 类）⚠️

D. 不可避免的固有矛盾（1 类）

LLM 训练截止日期 vs 回测时点 —— 这必须明文承认，不是 bug 是限制
3.2 主要矛盾在哪里？
按 contradiction-analysis 的思路看：当前系统的主要矛盾不是"哪个数据源接错了"，而是"effective_date 这条不变性在系统中没有任何全局守门人"。

具体表现：

数据采集层：函数级别自己决定要不要截断（铜金 ✓ / FGI ✗）
缓存层：完全不感知 effective_date，cache key 里有日期但内容不校验
artifact 层：根本没这个参数
评估层：DataIntegrity 不查时间一致性
这是架构问题，不是个别 bug。修个别 bug 治标不治本 —— 因为下次新增数据源时，没有架构强制让作者必须正确处理 effective_date。

3.4 老实话
这个系统在L1 / L5 / L3 universe 三个地方做了正确的 point-in-time 努力，说明设计者意识到了回测的严格性要求
但 L2 / Bridge / chart artifact / cache / DataIntegrity 这条链路上有大量"看起来对、实际穿越"的窟窿
这次 run 没产出污染报告纯属 L4 崩了挡了一刀，不能当作系统无前瞻的证据
距离"可信回测系统"的差距是结构性的，不是 bug-level 的。当下满足的原则数大约是 10 条里 3 条完全做到，3 条部分做到，4 条不同程度违反。
