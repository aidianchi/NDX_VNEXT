# CLAUDE.md

## 北极星

`ndx_vnext` 的目的是**做出可靠的 NDX 投资判断，并让判断质量随时间可测量地变好**。可审计推理链是手段，不是目的。

判断质量立在三根支柱上：**事实基础扎实**（每个数字经得起独立重算和来源追溯）、**证据面均衡**（多空两侧都有发言权，菜单不得先天偏科）、**校准闭环常开**（每个判断可评分、被事后评分，评分反过来修正方法）。

每个重要市场结论都要能回答六问：判断对象是什么 → 哪些证据有发言权 → 哪些确认它 → 哪些反驳它 → 什么情况需要改判 → 事后看对错如何、系统据此改了什么。

核心架构原则：**Context-first, role-second**。Agent 拆分的理由是上下文隔离和认知变换边界，不是模仿人类职位。

## 常驻边界

不可被普通待办覆盖，除非用户明确要求改变系统原则。

- **层间隔离**：L1-L5 运行时上下文不得互通。它们只能知道静态五层本体、投资对象定义和本层指标法典，不得接收其他层的运行时数据、摘要、结论、候选跨层关系，或 Bridge / Thesis 的当前判断。
- **主链纯度**：主要分析必须来自 vNext artifacts；legacy adapter 只做兼容导出和兜底映射。
- **冲突是资产**：不得为了报告顺滑而抹平跨层冲突、反证和未解决张力。
- **指标不得越权**：代理指标不能冒充官方事实，技术指标不能证明估值便宜，复合指标的子项不能绕过总分语义升格为强结论。
- **弱来源不进主链**：新闻、浏览器采集、登录态工具、sidecar 结果默认只是隔离观察或人工复核材料；未正式升级前不得进入 L1-L5 主证据链，不得成为 `evidence_ref`。
- **时点纪律**：回测与快照必须尊重有效日期。进入上下文的数据、新闻、图表行不得晚于回测日或 `effective_date`；缺口写入数据边界，不得用当前网页或联网搜索伪装成历史当时可见事实。
- **不得编造**：历史胜率、回测收益、样本期、概率数字、点位阈值——除非输入 evidence refs 明确提供，否则只能表达为待验证假设或风险观察。
- **发布闸门**：产物被标记 blocked / unpublishable 时，不得继续当作可发布结论处理。
- **历史材料不是现状**：`docs/archive/**` 与根目录带日期的旧审计文件记录的是当时的事实，可引用来说明"当时怎么判断"，不得当现状依据。

## 文档路由

按任务读最小必要文档，不要把所有背景塞进对话。

| 你要找什么 | 去哪 |
|---|---|
| 现在什么状态、该做哪件、什么在等用户 | `现在.md` —— **唯一可以声明事项状态的文件** |
| 用户否决过什么、在乎什么、为什么不做某事 | `人话进度报告.md` —— 决策记录，涉及方向与取舍时**必读** |
| 已完成的技术记录与验证结果 | `WORK_LOG.md`（只增不改，完成后按最新在上追加） |
| 某件事的调查细节、编号、施工方/验收方 | `investigation_reports/*/WORK_ORDERS.md` |
| 长期架构原则 | `ARCHITECTURE.md` |
| 指标判读、市场状态、跨层推理标准 | `RESEARCH_CANON.md` |
| 真实 run / 回测复盘流程 | `RUN_REVIEW_CHECKLIST.md` |
| 回测为什么必须那样做（第一性原理） | `回测原则.md` |
| 数据源覆盖、fallback、回测数据边界 | `DATA_COVERAGE_REVIEW.md` |
| 历史材料（不是现状） | `docs/archive/INDEX.md` |

冲突时：事项状态以 `现在.md` 为准；长期架构以 `ARCHITECTURE.md` 为准；已发生的完成记录以 `WORK_LOG.md` 为准；本文件的常驻边界优先于以上全部。

## 执行纪律

- 先判断、再最小修改、最后验证。不确定时先说清假设、歧义和取舍；合理假设会带来明显风险时，先向用户确认。
- 多条路径都可行时，选最简单、最贴近现有架构、最容易验证的一条。
- 改动外科手术式收敛：只改必需位置，匹配既有风格，不顺手重构，不清理无关旧代码。本次改动制造的未使用导入/变量/函数/文档残留必须同步清理，但不得删除本来就存在的无关死代码。
- 每个非平凡任务都要有可验证的成功标准。修 bug 要能复现并确认消失；改合约要有测试或 artifact 校验；改 UI 要能生成并检查目标页面。
- **状态只写一处**：未完成事项的**当前状态**只写在 `现在.md`。其他文档可以保留带日期的历史状态，但不得冒充现状。收工必须在同一次提交里：从 `现在.md` 删掉该行 + 在 `WORK_LOG.md` 用 `【关闭 T##】` 写完成记录 + 在工单末尾追加关单记录。编号永不复用。里程碑另加一封信。
- **信必须够格**：《人话进度报告》每封信至少包含"否决/未做及其理由"或"需要用户决定的事项"之一。仅把技术变更翻译成白话的信是副本，应删不应留。
- 跑 `pytest tests/test_docs_consistency.py` 自查文档纪律，不要靠记性。

## 常见误判

新对话容易把下面这些当成 bug 去"修"，都不是：

- 打分器返回"判断太年轻、无法评分" → 设计内的诚实跳过，第一批成熟样本要等 2026-07-27 之后。
- CFTC / 部分数据源标 unavailable → 本机网络限制下的诚实降级，不是遗漏，不得用别的源冒充补齐。
- 报告某节留空（如事件总结生成失败） → "宁缺毋滥"闸门在工作，先查 `quality_gate.notes` 有无留痕。
- 旧快照复核后仍判 blocked → 正确行为，修兼容不等于放松闸门。

## 协作偏好

- **回复分两版，顺序固定**：先"人话版"（面向非工程背景的决策者：结论、判断依据、影响范围、行动建议、剩余风险），再"技术版"（面向 AI agent 或核实者：代码引用、文件位置、实现细节、可执行核查线索）。人话版不是简化版，必须保留因果、取舍和证据链；技术版不是附录，必须能独立成立。只有一步操作、一个数字或一个明确状态时，可以只给人话版。
- 解释要通俗、准确、到位：少中英夹杂，少行业黑话。通俗不等于变浅。
- 复杂任务可多 agent 并行、用上下文隔离降低互相污染，但必须先说明拆分理由、各自边界和汇总方式。

## 验证原则

按改动范围选择，交付前必须说明验证结果；未做某项合理验证时说明原因和剩余风险。

- 代码合约与核心逻辑：跑相关单测，必要时全量（`.venv/bin/python -m pytest --cache-clear -q`）。
- 数据 / 回测 / 快照：检查 DataIntegrity、`run_summary.json`、`backtest_data_boundaries`、数据日期与发布状态。
- vNext artifact 或报告：用最新合适 run 生成 native `brief`，检查 evidence refs、冲突保留和审计区。
- workbench / console / UI：生成目标页面，做桌面与移动布局检查。
- 文档：跑 `tests/test_docs_consistency.py`。
