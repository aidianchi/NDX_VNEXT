# AGENTS.md

## 沟通

用户不默认懂软件工程。除极小文字修改外，工作时和交付时都要用通俗话说明：改什么、为什么、影响范围、验证结果和剩余风险。少用黑话；必须用术语时先翻译成人话。

## 回复结构

除非常简单的任务外，回复必须分两个版本，顺序固定：

1. 人话版在前：用非专业人员也能看懂的语言，把结论、判断依据和行动建议讲清楚。少中英夹杂，少行业黑话。
2. 技术版在后：给 AI agent 或需要深入核实的人看，包含代码引用、文件位置、具体实现细节。

人话版不是“简化版”：它同样要讲清因果、取舍和证据链，只是用更通俗的语言。技术版不是“附录”：它要能独立成立，让 agent 不依赖人话版就能执行。

如果任务非常简单（一步操作、一个数字、一个明确状态），可以只给人话版。

## 项目

`ndx_vnext` 生成可追问、可审计、可展开阅读的 NDX 投研推理链。核心原则：Context-first, role-second。

每个重要市场结论都必须说明：判断对象、支持证据、反驳证据、失效条件。

## 红线

- L1-L5 运行时上下文隔离；不得接收其他层运行时数据、摘要、结论、候选跨层关系，或 Bridge / Thesis 当前判断。
- 主分析来自 vNext artifacts；legacy adapter 只做兼容导出和兜底映射。
- 不得抹平冲突；反证、张力和未解决问题必须保留。
- 指标不得越权发言：代理指标不是官方事实，技术指标不能证明估值便宜，复合指标子项不能绕过总分语义。
- 新闻、浏览器、登录态工具和 sidecar 默认只是候选材料；未升级为正式数据源前，不得成为 L1-L5 主证据或 `evidence_ref`。
- 回测和快照必须尊重 `effective_date`；不得用当前信息伪装成历史当时可见事实。
- 不得编造历史胜率、回测收益、样本期、概率数字或点位阈值。
- DataIntegrity / publish status 是发布闸门；blocked / unpublishable 不能当作可发布结论。

## 按需读

- 当前状态与待办：`现在.md`（唯一状态源）
- 用户否决过什么、在乎什么：`人话进度报告.md`
- 命令入口：`README.md`
- 架构边界：`ARCHITECTURE.md`
- 指标标准：`RESEARCH_CANON.md`
- run / 回测复盘：`RUN_REVIEW_CHECKLIST.md`
- 数据边界：`DATA_COVERAGE_REVIEW.md`
- 完成记录：`WORK_LOG.md`
- 通俗解释风格：`docs/archive/2026-05/PLAIN_LANGUAGE_*.md`（历史样例，文风可借鉴、内容已过时）

冲突时：`ARCHITECTURE.md` 管长期原则，`现在.md` 管事项状态，`WORK_LOG.md` 管已完成事实；本文件红线优先。

## Codex 调度与成本控制

本项目使用 `.codex/config.toml` 和 `.codex/agents/` 的分层代理配置：根线程负责理解需求、拆分任务、整合结果和最终判断；子代理只负责边界清晰的局部工作。

- 可以并行委派给 `explorer`：文件搜索、代码路径定位、测试/日志读取、只读证据收集。
- 可以委派给 `worker`：小范围、机械性、可明确验收的修改和对应的窄范围验证。
- 涉及架构、投研结论、数据来源权威性、`effective_date`、L1-L5 隔离、DataIntegrity、`publish status`、反证或未解决冲突时，根线程必须自己判断；不得把最终判断交给低成本 worker。
- 有代码修改时，根线程应在整合前后调用 `reviewer` 做只读复核；reviewer 不能直接改文件。
- 委派必须给出明确的文件范围、输出格式和验收条件。子代理不得递归再委派，不能扩大任务范围，也不能覆盖用户已有未提交改动。
- 如果委派会增加上下文传递和返工成本，根线程可以不委派；省钱不是以牺牲证据链、审计性或发布闸门为代价。
